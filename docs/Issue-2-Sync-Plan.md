📋 [PRD 施工图]：mc-task-app 多端数据同步升级
1. 需求背景与核心目标
•	核心价值：解决“换设备数据丢失”问题，将单机版应用升级为云同步应用，提升用户粘性。
•	技术选型决策：
•	选择方案 A（Cloudflare D1 + Workers）的变体：既然你已经有阿里云服务器在跑后端（Express），最省钱且最稳妥的方案是直接利用你已有的阿里云后端 + 数据库（或者简单的 JSON 文件存储），这样不需要额外配置 Cloudflare 的海外环境。
•	折中方案：如果不想折腾后端数据库，先用 Firebase/Supabase 的国内镜像或替代品。
•	本 PRD 推荐：直接在阿里云后端增加一个简单的 Sync 接口，将数据存入服务器。
2. 涉及模块与文件清单
•	前端修改 (React)：
•	src/context/TaskContext.tsx 或 src/hooks/useLocalStorage.ts：修改数据读写逻辑，从“只读本地”改为“优先同步云端”。
•	src/services/api.ts（需新建）：定义上传/下载数据的接口。
•	后端修改 (Express)：
•	backend/server.js：增加 /api/sync 路由。
•	backend/data/user-data.json：临时作为“简易数据库”存储用户数据（适合初期，无需配置复杂数据库）。
3. 详细逻辑设计 (给“哈哈”和“阿二”看的)
•	匿名账号逻辑：
	1.	用户首次进入，生成一个 UUID 存入 localStorage。
	2.	提示用户：“记录你的唯一同步码”，用户在设备 B 输入该码即可同步。
•	同步策略 (Sync Strategy)：
•	上传：每当用户点击“完成任务”或“开启宝箱”时，前端在更新 localStorage 的同时，异步调用后端 API。
•	下载：应用初始化（App Mount）时，对比本地与云端的时间戳（lastSyncAt），取最新的覆盖旧的。

4. 质量验收标准 (给“阿一”看的)
	1.	场景模拟：在 Chrome 开发者工具里手动清除 localStorage，重新输入 userId 后，数据应能从 39.97.246.203 自动拉取恢复。
